# Hermes 特色与不同点系列规划 v1

## 这套系列现在为什么要改写法

在写完前两篇之后，方向已经更清楚了。

如果继续按最初那种“memory / recall / skills / compression 分主题平铺”的方式推进，当然也能写下去；但这样会有一个问题：

> **读者会知道 Hermes 有很多特色，却不一定立刻看清这些特色为什么成立、以及它们为什么和 coding agent 不一样。**

而第二篇已经把最关键的总抓手钉住了：

> **Hermes 的主循环更像经验处理内核，而 Codex / Claude Code 的主循环更像 coding work runtime。**

所以后面最自然的路线，不是再把“memory / recall / skill / compression”当成四个并列 feature 来讲，而是：

> **顺着主循环差异，专门拆 Hermes 那些最有辨识度的不同点，看它怎样把每个“不同”都接进经验闭环。**

这就是本系列规划从现在开始要切换的写法。

---

## 系列总问题

这套系列真正要回答的不是：

- Hermes 有哪些功能
- Hermes 和别家 agent 都有什么模块差异

而是更具体的这个问题：

> **如果把主循环当成比较起点，Hermes 到底在哪些地方和 coding agent 走了不同路线？这些不同路线，又怎样一步步把它推成一个会持续积累自己的系统？**

也就是说，后面每篇文章都应该同时回答两层：

1. **这一点和 coding agent 有什么不同**
2. **这个不同怎样服务于 Hermes 的自我进化闭环**

---

## 当前已完成的两篇

### 01｜为什么 Hermes 不是“有记忆的 agent”，而是“能持续积累自己的 agent”
作用：
- 先立全组入口
- 先把“自我进化”从 model-level learning 拉回 system-level growth
- 先把 4 条闭环压出来

### 02｜从主循环看，Hermes 与 coding agent 有什么不同
作用：
- 先把 Codex / Claude Code / Hermes 主循环做第一次结构性分野
- 先把后续所有文章的比较标准钉住
- 把 Hermes 定义成“经验处理内核”而不是“又一个 agent loop”

所以从现在开始，后续每篇都应该视为：

> **对第二篇里某一个“不同点”的深入展开。**

---

## 后续文章的展开原则

### 原则 1：每篇只抓一个“不同点”
不要一篇里又讲 memory、又讲 skills、又讲 compression。

一篇只回答一个问题，例如：
- 为什么 Hermes 会把持久记忆直接接进主循环
- 为什么 session recall 不是附属搜索
- 为什么 skills 不是插件市场，而是 procedural memory

### 原则 2：每篇都先对照 coding agent，再讲 Hermes
不要直接从 Hermes 自己讲起。

最好都先问：
- 如果用 Codex / Claude Code 的习惯看这里，会默认怎么理解？
- 为什么这个理解放在 Hermes 上会失效？
- Hermes 真正多出来的东西是什么？

### 原则 3：每篇都要回到“这如何服务于自我进化闭环”
后面每篇都不只是 feature 解释文。

必须明确回答：
- 这条机制到底让 Hermes 多保留了什么
- 多召回了什么
- 多沉淀了什么
- 多优化了什么

## 建议系列顺序

### 第 03 篇
**Hermes 到底把什么存下来了：从静态文件层看它怎样为自我进化准备长期材料**

#### 要回答的问题
- Hermes 除了主循环和运行时行为，还把哪些东西真正落到了磁盘上
- `MEMORY.md`、`USER.md`、`state.db`、skills 目录、trajectory 文件、`config.yaml` 分别在保存什么
- 为什么这些静态文件不是附属存档，而是 Hermes 自我进化机制的物质基础

#### 和第二篇的关系
第二篇已经说明 Hermes 的主循环更像经验处理内核。  
这一篇就专门拆：**既然它在处理经验，那这些经验最后到底以什么静态形态被留下来了。**

---

### 第 04 篇
**为什么 Hermes 会把持久记忆直接接进主循环，而不是把 memory 当附属功能**

#### 要回答的问题
- 为什么 `MEMORY.md / USER.md / frozen snapshot` 不只是一个 memory 实现细节
- 为什么 built-in memory provider / memory manager 会被直接接进主循环
- 为什么这条线在 Claude Code / Codex 里都没有被做成 Hermes 这种位置

#### 和第二篇的关系
第二篇已经说了：Hermes 的 loop 会先 prefetch experience，再进入当前回合。  
这一篇就专门拆：**为什么持久记忆会被放在这么前的位置。**

---

### 第 05 篇
**为什么 session recall 不是历史搜索，而是 Hermes 的第二层记忆系统**

#### 要回答的问题
- 为什么 `hermes_state.py + session_search_tool.py` 不能只理解成“聊天记录搜索”
- 为什么 FTS5 检索后还要再走 focused summary
- 为什么这层更像“情境回忆”，而不是固定 memory

#### 和第二篇的关系
第二篇说 Hermes 主循环会把过去经验先召回。  
这一篇就专门拆：**它到底怎么把“过去会话”变成当前可用经验。**

---

### 第 06 篇
**为什么 skill system 才是 Hermes 最像自我进化的地方**

#### 要回答的问题
- 为什么 skill 不是另一种 memory 文件
- 为什么 `skill_manager_tool.py` 里要明确区分 declarative memory 和 procedural memory
- 为什么 create / patch / edit / write_file 这些能力说明 skills 是可演化操作资产

#### 和第二篇的关系
第二篇说 Hermes loop 结束后不仅有结果，还有经验沉淀。  
这一篇就拆：**哪些经验最终会沉淀成之后可复用的做法。**

---

### 第 07 篇
**为什么 Hermes 不只是压缩上下文，而是在整理自己的过去**

#### 要回答的问题
- 为什么 `context_compressor.py` 不该只被看成 token 节约器
- 为什么 iterative summary update 很关键
- 为什么 prompt caching 与 compression 也是经验闭环的一部分

#### 和第二篇的关系
第二篇已经说 Hermes 会在长运行中继续处理经验。  
这一篇就专门拆：**经验为什么不会越积越重，而会被整理成还能继续工作的形式。**

---

### 第 08 篇
**为什么 Hermes 的主循环收口后还没结束：background review 才是经验沉淀的关键转折**

#### 要回答的问题
- 为什么 turn 结束后还会有 `sync_all`、`queue_prefetch_all`、background review
- 为什么这一步比普通 agent 的“输出完成”更像 post-hoc reflection
- 为什么这一步是 Hermes 最不同于 coding agent 的地方之一

#### 和第二篇的关系
第二篇已经说明 Hermes loop 和 coding agent loop 的不同。  
这一篇会进一步把这个差异推到最锋利：**Hermes 的主循环结束后，系统还在继续思考“什么值得留下”。**

---

### 第 09 篇（收束篇）
**Hermes 怎样把 memory、recall、skills、compression 接成一套真正的 self-evolution loop**

#### 要回答的问题
- 前面几篇拆开的不同点，最后怎样重新接回 `run_agent.py`
- 为什么 Hermes 的成长更像 system-level growth，而不是 model-level growth
- 为什么 Hermes 的真正特色不是模块多，而是这些模块的连接方式不同

#### 和第二篇的关系
第二篇是第一次总对比。  
这一篇是最后一次总收束：把系列重新压回一条总判断。

---

## 系列地图（当前推荐）

1. 入口篇：Hermes 为什么不是“有记忆的 agent”
2. 主循环比较篇：Hermes 与 coding agent 有什么不同
3. 静态文件篇：Hermes 到底把什么存下来了
4. 持久记忆篇：为什么 memory 被直接接进主循环
5. 会话回忆篇：为什么 session recall 是第二层记忆系统
6. 技能沉淀篇：为什么 skills 是 procedural memory
7. 上下文整理篇：为什么 compression / caching 是经验整理系统
8. 反思沉淀篇：为什么 background review 是关键转折
9. 收束篇：Hermes 怎样形成真正的 self-evolution loop

---

## 每篇都应该保持的写法

后面系列统一保持这套写法：

1. **先从 coding agent 的默认理解切入**
2. **再指出这个理解在 Hermes 上为什么不够**
3. **再用源码锚点证明 Hermes 真正的不同点**
4. **最后回到：这个不同怎样服务于自我进化闭环**

这样系列的节奏就不会散。

---

## 最后要留下的一句话

> **后面这组文章，不是把 Hermes 的特色一个个罗列出来，而是顺着主循环差异，把它那些最关键的不同点逐个拆开，最后重新收成一套 self-evolution loop。**
