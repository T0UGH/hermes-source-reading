## 问题记录

### Q001
- **时间**：2026-04-18
- **阅读位置**：第五篇《session recall 为什么不是历史回放，而是按需重构过去经验》
- **问题**：session recall 是主循环中的一部分吗？还是它只是一个 tool，需要触发 tool call 才会被调？
- **即时回答摘要**：它不是像 memory prefetch 那样“每轮默认接进主循环”的能力；更准确地说，session recall 的**数据沉淀**是主循环持续在做的，因为 `run_agent.py` 会把会话消息持续刷进 `SessionDB`；但“把过去会话拿出来检索/总结”这一步，是通过 `session_search` 工具完成的，需要模型在当前轮显式发起 tool call 才会真正执行。所以它属于“主循环已接好基础设施，但检索动作按需触发”，不是每轮自动召回。
- **是否需要后续展开**：可继续补一张“会话写入 vs 会话召回”的边界图

### Q002
- **时间**：2026-04-18
- **阅读位置**：第五篇《session recall 为什么不是历史回放，而是按需重构过去经验》延伸问题
- **问题**：如果把这套机制引入 Claude，是不是通过 MCP 注册一个 session search tool 比较好？
- **即时回答摘要**：如果目标是“先把按需 recall 能力接进去”，那通过 MCP 暴露一个 `session_search` 工具是最合适、最小侵入的第一步，因为 Claude 很擅长在需要时主动调用工具；但如果目标是复刻 Hermes 这套完整机制，只做一个 MCP tool 还不够。Hermes 真正完整的是两层：第一层是主循环外/主循环旁边持续做会话归档与索引，第二层才是模型在当前轮按需调用 `session_search`。所以对 Claude 更准确的建议是：**MCP tool 适合做 recall front door；后台还需要一套独立的 transcript 持久化与索引层**。否则你只有“能搜”的接口，没有“持续积累经验池”的底座。
- **是否需要后续展开**：可继续补“Claude 接入版最小实现 / 完整实现”两档方案

### Q003
- **时间**：2026-04-18
- **阅读位置**：第六篇《为什么 skill system 才是 Hermes 最像自我进化的地方》
- **问题**：如果不是 Hermes 自己总结的 skill，而是从外部下载的 skill，Hermes 也会编辑它吗？
- **即时回答摘要**：会，至少从当前源码实现看是会的。`skill_manage` 在编辑、patch、写 supporting file、删除 skill 时，不区分“这是自己生成的 skill”还是“外部下载的 skill”，而是统一通过 `_find_skill(name)` 去所有 skill 目录里找；而 `_find_skill()` 又会调用 `get_all_skills_dirs()`，搜索顺序是本地 `~/.hermes/skills/` 在前，外部 `skills.external_dirs` 在后。也就是说，只要这个外部 skill 在 Hermes 的 skill 目录搜索范围里、并且底层目录可写，Hermes 就能直接 patch / edit 它。真正的边界不是“来源”，而是“是否被纳入技能目录、是否能写”。
- **是否需要后续展开**：可继续补“外部 skill / 本地 skill / 同名覆盖”三者优先级
